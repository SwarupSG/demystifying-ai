The Collaboration Framework

AI is a collaborator, not a vending machine. Work in steps, not wishes.

1
Set Role
"You are a [role] helping me..."
2
Give Context
"This is for [audience/purpose]..."
3
Set Constraints
"Do NOT... / Build ONLY... / STOP..."
4
Build One Thing
Request one feature, verify it works
5
Repeat
Next feature only after confirmation

The SLC Framework

Not "minimum viable" — Simple, Lovable, Complete. Build in this order.

SIMPLE
40%
Core function only
Does it WORK?
LOVABLE
35%
Emotional resonance
Do they WANT to use it?
COMPLETE
25%
Reliability & polish
Does it handle failure?

Prompt Templates

Copy these templates and customize for your project.

Starting a Project:
You are a [ROLE] helping me build [PROJECT]. CONTEXT: This is for [AUDIENCE/PURPOSE]. [KEY CONSTRAINT, e.g., "Speed matters more than features."] CONSTRAINTS: – Do NOT add features I haven't requested – Do NOT suggest alternatives unless asked – After each step, STOP and wait for confirmation SCOPE: This project includes ONLY: 1. [Component 1] 2. [Component 2] 3. [Component 3] Build step-by-step. Confirm each step before continuing. Say "Ready" and wait.
When Something Goes Wrong:
The [element] isn't working correctly. Expected: [what you asked for] Actual: [what happened] Fix ONLY this issue. Do not change anything else.
When AI Adds Unwanted Features:
You added [feature] which I didn't request. Remove it completely. Keep everything else as is.

Key Phrases

Instead of... Say...
"Make it good" "Use [specific style/format]"
"Add some features" "Add ONLY [specific feature]"
"Fix it" "Expected X, got Y. Fix only this."
"Keep going" "STOP after this step. Confirm."
(nothing) "Do NOT [thing you don't want]"

Complete Example: Hydration Tracker App © Proprietary

⚠️ © 2025 Swarup Biswas. This worked example and SLC build methodology are proprietary. For personal reference only. View full terms.

A worked example showing all 11 prompts to build a complete app using the SLC framework. Click each step to expand the full prompt.

Step 0 — Initialization SETUP
Teaching Point: Set strict boundaries BEFORE building. This prevents AI from improvising.
You are a senior frontend developer helping me build a simple hydration tracking web app using the SLC framework (Simple → Lovable → Complete). CONTEXT: This app is for busy professionals who forget to drink water. Speed and simplicity matter more than features. We're building this in under 90 minutes. IMPORTANT CONSTRAINTS: – Do NOT add features I have not explicitly requested – Do NOT suggest alternative layouts or screens unless asked – Build each step EXACTLY as described – After completing each step, STOP and wait for the next prompt – If you're unsure about anything, ASK rather than guessing UI REQUIREMENTS: – Mobile-first, responsive design – Minimum 48px touch targets for all buttons – Clean, minimal layout with system fonts – Blue/white color theme SCOPE: The app requires ONLY the following screens: 1. Home (main tracking screen) 2. Settings 3. Onboarding (3-step carousel) 4. Help / FAQ 5. Goal Celebration modal I will prompt you step-by-step through SIMPLE → LOVABLE → COMPLETE phases. Confirm when each step is done. Say "Ready for Step 1" and wait.
Expected Response: AI confirms understanding and says "Ready for Step 1"
Step 1 — Base Screens SIMPLE
Teaching Point: Structure before substance. Create the skeleton first, no logic yet.
Create ONLY the screens listed below, nothing else: – Home – Settings – Onboarding (3-step carousel) – Help / FAQ – Goal Celebration modal Requirements: – Minimalist blue/white theme – Clean layout with clear visual hierarchy – Clear navigation: Home → Settings → Help – Do NOT add any features or logic yet – Only create screen structure + headings EXCLUSIONS: No water logging, no progress bar, no avatar, no animations yet. STOP after screen creation. Confirm with: "Base screens created. Ready for Step 2."
Verify: All 5 screens exist with headings. Navigation works. No functionality yet.
Step 2 — One-Tap Logging SIMPLE
Teaching Point: The core feature, nothing else. This is the heart of SIMPLE.
On the Home screen, add ONLY these elements: 1. Three large buttons labeled: +100 ml, +200 ml, +300 ml 2. A numeric display showing today's total water intake Logic Requirements: – When a button is tapped, increase hydrationTotal by the corresponding amount – Store updated hydrationTotal in LocalStorage – Display updated total immediately on screen – Data must persist across page refreshes EXCLUSIONS: NO avatar, NO animations, NO progress bar yet. STOP after implementing ONLY this functionality. Confirm with: "One-tap logging implemented. Ready for Step 3."
Verify: Tap each button → total increases. Refresh page → total persists.
Step 3 — Progress Bar + Reset SIMPLE
Teaching Point: Visual feedback creates behavior loops. Still functional, not delightful yet.
Add a progress bar to the Home screen that visualizes: hydrationTotal / dailyGoal Behavior: – Progress bar updates immediately on each log – Color ranges: • 0–40% = grey • 40–80% = light blue • 80–100% = bright blue – Implement auto-reset at midnight using lastSavedDate stored in LocalStorage – When date changes, reset hydrationTotal to 0 Default dailyGoal: 2000 ml EXCLUSIONS: Do NOT add avatar or celebrations yet. STOP after the progress bar + reset logic is functional. Confirm with: "Progress bar + reset logic complete. Ready for Step 4."
Verify: Progress bar fills as you log. Colors change at thresholds. Change device date → total resets.
Step 4 — Progressive Reminders SIMPLE
Teaching Point: Build the mechanism before the personality. Reminders work, but aren't charming yet.
Implement progressive reminders based on inactivity. Behavior Requirements: – Track timestamp lastLogTime whenever user logs water – Trigger notifications at: 15 min, 30 min, 45 min, 60 min of inactivity – Use the browser Notification API – Reminders MUST stop after the user logs water again – Request notification permission on first use UI Requirements: – Add a simple toggle in Settings: "Reminders ON/OFF" – Save toggle state in LocalStorage – Only send notifications when toggle is ON EXCLUSIONS: Do NOT add micro-copy or contextual messages yet. Generic reminder text is fine. STOP after implementing basic reminders + toggle. Confirm with: "Progressive reminders implemented. Ready for Step 5."
Verify: Enable reminders → wait 15 min → notification appears. Log water → timer resets. Toggle off → no notifications.
Step 5 — Daily Goal Settings SIMPLE
Teaching Point: SIMPLE is now complete. It WORKS, but you wouldn't ENJOY using it yet.
On the Settings screen, add: 1. Numeric input field for dailyGoal – Default: 2000 ml – Range: 500–5000 ml – Include + / - buttons for easy adjustment 2. Toggle for reminders (linked to Step 4) 3. Store all values in LocalStorage 4. Ensure Home progress bar uses the updated goal immediately EXCLUSIONS: Do NOT add UI decoration, animation, or advanced settings. STOP after Settings is fully functional. Confirm with: "Settings implemented. SIMPLE phase complete. Ready for LOVABLE Step 6."
Verify: Change goal → progress bar recalculates. Refresh → settings persist. All SIMPLE features work together.
Step 6 — Avatar LOVABLE
Teaching Point: Now we add personality. This transforms "tool" into "companion."
Add a hydration avatar to the Home screen. Requirements: – Avatar has ONLY 3 states: • Neutral face: 0–40% • Content face: 40–80% • Happy face: 80–100% – Avatar must update dynamically when progress changes – Add a subtle bounce animation on each water log – Keep avatar style neutral, adult-friendly (no cartoons, no childish designs) – Position avatar prominently but not distractingly EXCLUSIONS: No sounds, no complex animations, no multiple avatar options. STOP after avatar is added and functional. Confirm with: "Avatar implemented. Ready for Step 7."
Verify: Log water → avatar bounces. Cross 40% → face changes. Cross 80% → face changes again.
Step 7 — Celebration Modal LOVABLE
Teaching Point: The payoff moment. This is the emotional reward that builds habits.
Implement a goal-completion celebration modal that appears when: hydrationTotal >= dailyGoal Requirements: – Modal text: "Goal achieved! Your body thanks you." – Add subtle confetti animation (CSS-based, no heavy libraries) – Add "Continue" button to dismiss modal – Ensure celebration triggers ONLY once per day (track in LocalStorage) – Modal should feel rewarding but not over-the-top EXCLUSIONS: Do NOT add sound effects. STOP after modal + confetti are functional. Confirm with: "Goal celebration implemented. Ready for Step 8."
Verify: Hit goal → modal appears with confetti. Dismiss → continue using app. Refresh → modal doesn't reappear same day.
Step 8 — Micro-copy LOVABLE
Teaching Point: Words create personality. The same function feels different with thoughtful copy.
Enhance reminder notifications with thoughtful micro-copy. Rotate through ONLY these 5 messages: 1. "A quick sip helps your focus." 2. "Small hydration boost?" 3. "Your body could use a refill." 4. "Long meeting? A sip may help." 5. "Stay refreshed, stay sharp." Requirements: – Rotate randomly through messages (not sequential) – Must respect the reminder ON/OFF toggle – Do NOT over-notify (keep existing timing intervals) EXCLUSIONS: No additional notification types, no custom message creation. STOP after messages are integrated. Confirm with: "Micro-copy integrated. LOVABLE phase complete. Ready for COMPLETE Step 9."
Verify: Trigger multiple reminders → messages vary. Messages feel human, not robotic.
Step 9 — Error Handling COMPLETE
Teaching Point: COMPLETE means handling failure gracefully. The app should never leave users confused.
Add clear error handling and permission management: 1. If notification permission denied: – Show in Settings: "Reminders are off. To enable, allow notifications in your browser settings." – Disable the reminder toggle with explanation 2. If LocalStorage is unavailable: – Show toast/banner: "We couldn't save your progress. Check if private browsing is enabled." – App should still function for current session 3. If reminder scheduling fails: – Show: "Reminder couldn't be scheduled. Please try again." – Provide retry option Requirements: – Error messages should be helpful, not technical – Never show raw error codes to users – Always provide a path forward (what to do next) STOP after error handling is complete. Confirm with: "Error handling complete. Ready for Step 10."
Verify: Deny notification permission → helpful message appears. Open in private mode → appropriate warning.
Step 10 — Onboarding + FAQ COMPLETE
Teaching Point: First impressions matter. Onboarding sets expectations and builds confidence.
Create a 3-step onboarding carousel: Page 1: "Tap to log your water." (show button visual) Page 2: "Watch your progress rise." (show progress bar visual) Page 3: "Get gentle reminders when needed." (show notification visual) Requirements: – Add Skip button (top right) – Add "Get Started" button on final page – Show onboarding only on first visit (track in LocalStorage) – Allow re-accessing onboarding from Help screen Help/FAQ Modal Content: – "Why track water?" - Brief health benefits – "How do reminders work?" - Explains progressive timing – "Where is my data stored?" - Explain LocalStorage, privacy-friendly, no server – "How do I reset my data?" - Instructions to clear STOP after onboarding + FAQ are functional. Confirm with: "Onboarding + FAQ complete. App build finished!"
Verify: Clear LocalStorage → onboarding appears. Complete onboarding → goes to Home. Help screen shows all FAQ items.

Your First Win

Before You Start, Answer These:

  • What's one task you do repeatedly that feels mechanical?
    Not complex. Just tedious.
  • What would "good enough" look like?
    Not perfect. Just useful to you.
  • When will you spend 90 minutes on this?
    Actual day and time in the next 7 days.

Your Commitment Device

A: Calendar Block 90 min now.
Label: "First AI Build"
B: Partner Tell someone:
"I'm building X by [date]"
C: Future Email Email yourself for 7 days:
"Did you build X?"
The 15-Minute Rule: When you don't feel like starting, commit to just 15 minutes. Set a timer. What actually happens: you don't stop. Starting is the hard part.

First Project Criteria

  • Small — 2-3 hours, not weeks
  • Personal — for YOU, not a company initiative
  • Low stakes — if it fails, nothing bad happens
  • Immediately useful — you'd actually use the output

If you can't check all four, pick a smaller project.

The 80/20 Reality

AI builds ~80%: structure, first drafts, functionality.

You provide ~20%: judgment, refinement, edge cases.

That's leverage, not limitation.
